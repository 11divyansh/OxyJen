# What's New in v0.4

## 1. Typed LLM Outputs

You can now define:

``` java
SchemaNode<User>
```

...and receive a `User` object directly, not a string.

-   No manual parsing.
-   No fragile regex.
-   No unsafe casting.

------------------------------------------------------------------------

## 2. JSON Schema Generation from Java Classes

Generate a JSON schema automatically:

``` java
JSONSchema schema = SchemaGenerator.fromClass(User.class);
```

Supports:

-   Records
-   POJOs (getter-based)
-   Nested objects
-   Lists / Sets / Arrays
-   Maps\<String, V\>
-   Enums
-   Optional`<T>`
-   Validation annotations

------------------------------------------------------------------------

## 3. Built-in JSON Parser

Oxyjen now includes a lightweight JSON parser:

``` java
Object parsed = JsonParser.parse(jsonString);
```

Parses into:

 | JSON    |  Java |
 | --------| --------------------------------| 
 | object  |  Map\<String,Object\> |
 | array   |  List`<Object>` |
 | string  |  String |
 | number  |  BigDecimal / Long / BigInteger |
 | boolean |  Boolean |
 | null    |  null |

------------------------------------------------------------------------

## 4. JSON -> Java Object Mapping

Convert validated JSON directly into Java types:

``` java
User user = JsonMapper.deserialize(json, User.class);
```

Supports:

-   Records
-   POJOs with setters
-   Nested objects
-   Collections
-   Arrays
-   Maps
-   Enums
-   Optional fields

------------------------------------------------------------------------

## 5. Schema Validation Engine

Validate model output against schema:

``` java
SchemaValidator validator = new SchemaValidator(schema);
var result = validator.validate(json);

if (!result.isValid()) {
    System.out.println(result.formatErrors());
}
```

Detects:

-   Missing required fields
-   Unknown fields
-   Type mismatches
-   Pattern violations
-   Enum mismatches
-   Numeric constraints
-   Nested errors

------------------------------------------------------------------------

## 6. New SchemaNode`<T>`{=html}

SchemaNode combines:

-   LLM call
-   Retry loop
-   Schema validation
-   JSON parsing
-   Java object mapping

### Example

``` java
SchemaNode<User> node = SchemaNode.builder(User.class)
    .model("gpt-4o-mini")
    .schema(SchemaGenerator.fromClass(User.class))
    .maxRetries(3)
    .memory("chat")
    .build();

User user = node.process("Generate a user named Alice aged 30", context);
```

Result:

    User[name=Alice, age=30]

# Example Usage

## Step 1 --- Define your Java type

### Record example

``` java
@Description("User profile")
public record User(
    @Description("User name")
    @Size(min = 1, max = 50)
    String name,

    @Min(0)
    @Max(150)
    int age
) {}
```

------------------------------------------------------------------------

## Step 2 --- Generate schema

``` java
JSONSchema schema = SchemaGenerator.fromClass(User.class);
System.out.println(schema.toJSON());
```

------------------------------------------------------------------------

## Step 3 --- Build SchemaNode

``` java
SchemaNode<User> node = SchemaNode.builder(User.class)
    .model("gpt-4o-mini")
    .schema(schema)
    .maxRetries(3)
    .build();
```

------------------------------------------------------------------------

## Step 4 --- Run

``` java
User result = node.process(
    "Create a JSON user with name Bob and age 42",
    context
);
```

------------------------------------------------------------------------

## Step 5 --- Use typed result

``` java
System.out.println(result.name());
System.out.println(result.age());
```

-   Deterministic, observable execution

## Architecture

ToolCall → ToolNode → ToolExecutor → Tool → ToolResult

------------------------------------------------------------------------

# Public API Layer

## Tool Interface

Core abstraction for any executable unit. Interface acting as a building block of more tools.

``` java
public interface Tool {
    String name();
    String description();
    JSONSchema inputSchema();
    JSONSchema outputSchema();
    ToolResult execute(Map<String, Object> input, NodeContext context);
}
```

------------------------------------------------------------------------

## ToolCall

Represents a request to call a tool. Takes in name of the tool and arguments.

``` java
ToolCall call = ToolCall.of("file_read", Map.of(
            "path", "/tmp/test.txt",
            "offset", 5
        ));    
```

------------------------------------------------------------------------

## ToolResult

Result of a tool execution, tell success/failure, output, error, metadata.
``` java
ToolResult result = executor.execute(call, context);
result.isSuccess();
result.getOutput();
result.getError();

ToolResult.success()
ToolResult.failure()
```

------------------------------------------------------------------------

## ToolExecutor

1. Tool Registry – Stores tools in an immutable map, prevents duplicates
2. Input Validation – Validates args using JSON schema (strict mode supported)
3. Permissions – Allows/block tools via ToolPermission
4. Safety Check – Runs tool.isSafe() before execution
5. Sandbox Execution – Enforces timeout & safe execution (sandbox.execute(() -> tool.execute(...)))
6. Output Serialization – Converts output to JSON-friendly format
7. Output Validation – Optional schema validation for outputs
8. Execution Tracking – Measures time + logs everything
9. Fail-safe Design – Never throws, always returns ToolResult
10. Lifecycle Hooks – Pre/post execution hooks via permissions(beforeExecution/afterExecution)
11. Builder Support – Flexible configuration

```java
ToolExecutor executor = ToolExecutor.builder()
    .addTool(new FileReaderTool(sandbox))
    .strictInputValidation(true)
    .validateOutput(true)
    .sandbox(sandbox)
    .permission(permission)
    .build();
```

 ------------------------------------------------------------------------

## ToolNode

ToolNode is a graph node wrapper around ToolExecutor.

1. Executes tools inside a graph
2. Delegates execution to ToolExecutor
3. Acts as a NodePlugin (fits into graph pipeline)
4. Supports multiple tools in one node
5. Provides lifecycle logging (onStart, onFinish, onError)
6. Exposes helper methods (hasTool, getToolNames)

```java
ToolNode toolNode = new ToolNode(
    new FileReaderTool(sandbox),
    new HttpTool(...)
);

Graph workflow = GraphBuilder.named("agent-pipeline")
    .addNode(routerNode)
    .addNode(toolNode)
    .addNode(summaryNode)
    .build();
```

## Validation

- What it Validates?
1. Core Checks
2. ToolCall is not null
3. Tool is not null
4. Tool name matches (call.name == tool.name)
5. Arguments match JSON schema
6. Required fields are present
7. Types are correct

- Strict Mode (Important)
  If strictMode = true:
  Rejects unknown arguments
  Prevents LLM hallucinated inputs
  ToolValidator validator = new ToolValidator(true);

- Warnings
If no schema -> warning (not failure)
Non-critical issues are tracked separately


```java
JSONSchema schema = JSONSchema.object()
    .property("a", PropertySchema.number("first number").build())
    .property("b", PropertySchema.number("second number").build())
    .required("a", "b")
    .build();

ToolCall call = ToolCall.of("calculator", Map.of(
    "a", 5,
    "b", 3
));

ToolValidator validator = new ToolValidator(true);
ValidationResult result = validator.validate(call, tool, context);

result.isValid();
```

# Safety Layer

## ToolPermission

Basic interface to make your own permission systems
``` java
// Controls WHO can use WHICH tools
if (!permission.isAllowed("file_delete", context)) {
    return blocked;
}
```

## AllowListPermission

Simple allowlist based permission system
``` java
// Simple implementation: explicit allow list
AllowListPermission.allowOnly()
    .allow("calculator")
    .allow("web_search")
    .build();
```

## ToolSandbox

Sandboxes tool execution to prevent escape and limit damage. Comes with two modes basic() and strict()(for strict validation).
``` java
private Path tempDir = Files.createTempDirectory("filereader-test-");
ToolSandbox sandbox = ToolSandbox.builder()
            .allowedDirectory(tempDir.toString())
            .timeout(5, java.util.concurrent.TimeUnit.SECONDS)
            .build();
```

# Built-in Tools

## FileReaderTool

Overview

FileReaderTool is a secure, flexible file reading tool designed for safe access to local files with advanced capabilities like chunking, partial reads, and metadata extraction.

Key Features
Sandboxed file access
- Uses ToolSandbox to restrict file paths
- Prevents directory traversal & unauthorized access
  
Metadata support
- File size, MIME type, last modified timestamp
  
Partial reading
Read using:
- offset + limit (byte-based)
- lineStart + lineEnd (line-based)

Chunked reading
- Efficient for large files
- Supports chunkSize + chunkIndex
  
Caching
- Avoids repeated disk reads for unchanged files
  
Binary mode
- Returns Base64 encoded content for binary-safe handling
  
Size limits
- Prevents reading excessively large files

```java
Tool fileTool = new FileReaderTool(sandbox);

ToolCall call = ToolCall.of("file_read", Map.of(
    "path", "/tmp/data.txt",
    "lineStart", 1,
    "lineEnd", 10
));

ToolResult result = executor.execute(call, context);
```


## HttpTool

Overview
HttpTool is a safe HTTP client tool for making API calls and fetching web data with built-in security and limits.

Key Features
Supports HTTP methods
- GET, POST, PUT, DELETE, PATCH
  
Domain allowlist
- Restrict requests to trusted domains only
  
Timeout control
- Prevents hanging requests
  
Response size limiting
- Avoids large/unbounded payloads
  
Headers & query support
- Custom headers
- Query parameters

```java
Tool httpTool = HttpTool.builder()
    .allowDomain("api.github.com")
    .timeout(5000)
    .build();

ToolCall call = ToolCall.of("http_request", Map.of(
    "method", "GET",
    "url", "https://api.github.com/users/octocat"
));

ToolResult result = executor.execute(call, context);
```

# 🧪 Example

## HttpTool example - Create a github issue with HttpTool
``` java
public static void main(String[]args) {
		HttpTool httpTool = HttpTool.builder()
				.allowDomain("api.github.com")
				.timeout(5000)
				.maxResponseSize(100000)
				.build();
		String token = System.getenv("GITHUB_TOKEN");
		Map<String, Object> input = new HashMap<>();
		input.put("method", "POST");
		input.put("url", "https://api.github.com/repos/11divyansh/EventManagementSystemAPI/issues");
		input.put("headers", Map.of(
		    "Authorization", "Bearer " + token,
		    "Accept", "application/vnd.github+json",
		    "Content-Type", "application/json"
		));

		input.put("body", """
		{
		  "title": "Test HttpTool issue creation",
		  "body": "Something broke in production"
		}
		""");
		NodeContext context = new NodeContext();
		ToolResult result = httpTool.execute(input, context);
		if (result.isSuccess()) {
		    System.out.println("Issue created!");
		    out.println(result.getOutput());
		    out.println(result);
		    out.println(result.getMetadata("location", String.class));
		} else {
			out.println("Error: " + result.getError());
		    result.getHttpStatus().ifPresent(status -> {
		        if (status == 404) {
		            out.println("Resource not found (fallback?)");
		        } else if (status >= 500) {
		            out.println("Server error (retry?)");
		        }
		    });
		    result.getHttpError().ifPresent(err -> {
		        out.println("Error Type: " + err.get("_errorType"));
		    });
		    result.getMetadata("headers", Map.class).ifPresent(headers -> {
		        out.println("Headers: " + headers);
		    });
		    result.getHttpMessage().ifPresent(out::println);
		}
	} 
```
