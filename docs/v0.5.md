
# Oxyjen v0.5 Documentation

## Deterministic Graph Execution Runtime for AI Workflows

Oxyjen v0.5 moves the project from "LLM helper utilities" toward a deterministic execution runtime for Java. The important idea is simple:

> Model calls, tools, validators, routers, retries, joins, and aggregations are all explicit nodes in a graph.

That makes AI workflows easier to test, debug, replay, observe, and reason about. Instead of hiding behavior inside an agent loop, Oxyjen exposes the execution plan as a graph of nodes and typed edges.

This document focuses on the v0.5 graph module and the core classes it depends on.

---

## v0.5 Class Map

Core:

- `NodePlugin<I, O>`: node contract.
- `NodeContext`: execution-scoped state, metadata, memory, logger, runtime.
- `Graph`: graph topology.
- `GraphBuilder`: fluent DSL for graph creation.
- `Edge`: base transition abstraction.

Execution:

- `ParallelExecutor`: graph executor.
- `ExecutionRuntime`: concurrency, timeout, failure mode, metrics.
- `DAGValidator`: structural graph validation.

Edges:

- `DirectEdge`: always traverse on success.
- `ConditionalEdge<O>`: traverse on predicate.
- `FailureEdge`: traverse on failure.
- `CyclicEdge`: bounded loop.

Branching:

- `BranchNode<I>`: mutually exclusive routing.
- `RouterNode<I>`: multi-route fan-out.
- `MergeNode`: fan-in and merge.

Concurrency and aggregation:

- `ParallelNode<I, O>`: fixed parallel tasks over one input.
- `MapNode<I, O>`: same task over many input items.
- `GatherNode`: collect/filter/sort/group/aggregate results.
- `GatherCollectors`: collection helpers used by `GatherNode`.
- `CollectionMode`: controls which task results are gathered.
- `TaskResult<T>`, `Success<T>`, `Failure<T>`, `Cancelled<T>`, `NotExecuted<T>`: deterministic task outcome model.

Schema integration:

- `SchemaNode<T>`: LLM + schema validation + typed Java output.
- `SchemaResult`: raw JSON + validity + field errors.
- `FieldError`: structured validation error.

## Mental Model

An Oxyjen graph has four main concepts:

| Concept | Class | Purpose |
| --- | --- | --- |
| Node | `NodePlugin<I, O>` | A unit of work. Takes an input and returns an output. |
| Edge | `Edge` | A transition rule from one node to another. |
| Graph | `Graph` | The topology: nodes plus edges. |
| Runtime | `ParallelExecutor` + `ExecutionRuntime` | Executes the graph with concurrency, failure mode, and timeout behavior. |

The graph runtime is intentionally explicit:

- If a path branches, you model that with `BranchNode`, `RouterNode`, or `ConditionalEdge`.
- If parallel paths must join, you model that with `MergeNode`.
- If a batch must run concurrently, you use `MapNode` or `ParallelNode`.
- If errors should route somewhere, you use `FailureEdge`.
- If a loop is intended, you use `CyclicEdge` with a hard max iteration cap.

## Which Feature Should I Use?

Several graph features overlap by design. This table is the quickest way to choose the right one.

| Need | Use | Why |
| --- | --- | --- |
| Simple A -> B pipeline | `connect("A", "B")` / `DirectEdge` | Always run the next node after success. |
| One of many paths, like if / else-if / else | `BranchNode` | Mutually exclusive routing. First matching branch wins. |
| Multiple paths can fire from the same input | `RouterNode` | Fan-out to all matching routes. |
| Conditional transition based on output or context | `connectConditional(...)` / `ConditionalEdge` | Edge-level branching without adding a router node. |
| Route failures to a recovery or merge node | `connectOnFailure(...)` / `FailureEdge` | Keeps success and failure transitions separate. |
| Route any node failure to one handler | `connectAnyFailureTo(...)` / `connectOnAnyFailure(...)` | Build-time failure policy for unknown failing sources. |
| Join several graph branches | `MergeNode` | Explicit fan-in with success/error collection and timeout. |
| Run a fixed set of tasks inside one node | `ParallelNode` | Encapsulated fan-out/fan-in, simpler graph topology. |
| Run the same function over many input items | `MapNode` | Batch mapping with order preservation and per-item results. |
| Collect/filter/sort/aggregate outputs | `GatherNode` | Normalizes results from `ParallelNode`, `MapNode`, iterables, or single values. |
| Retry/refine loop with a hard stop | `CyclicEdge` via `repeat()` / `loop()` | Intentional cycle with max iterations. |

Rule of thumb:

- Use `BranchNode` when only one path should run.
- Use `RouterNode` when several paths may run in parallel.
- Use `ConditionalEdge` when the condition belongs to the edge, not to a named routing node.
- Use `ParallelNode` when parallelism is an implementation detail inside one node.
- Use `RouterNode` + `MergeNode` when parallelism is part of the visible workflow topology.
- Use `MapNode` for "same operation over N items".
- Use `GatherNode` after `MapNode` or `ParallelNode` when you want clean downstream aggregation.

## Core API

### `NodePlugin<I, O>`

`NodePlugin` is the base contract for every node in Oxyjen.

```java
public interface NodePlugin<I, O> {
    O process(I input, NodeContext context);

    default String getName() {
        return this.getClass().getSimpleName();
    }

    default void onStart(NodeContext context) {}
    default void onFinish(NodeContext context) {}
    default void onError(Exception e, NodeContext context) {}
    default NodePlugin<?, ?> unwrap() { return this; }
}
```

Example:

```java
class NormalizeText implements NodePlugin<String, String> {
    @Override
    public String process(String input, NodeContext context) {
        return input.trim().toLowerCase();
    }

    @Override
    public String getName() {
        return "normalize";
    }
}
```

When to use:

Use `NodePlugin` for all custom workflow steps: LLM calls, API calls, validation, transformation, persistence, tool calls, scoring, guardrails, or routing preparation.

Note:

Prefer small deterministic nodes. A node should do one thing and return a value that downstream nodes can inspect.

---

### `NodeContext`

`NodeContext` is shared across the graph execution. It stores data, metadata, scoped memories, loggers, exception handlers, gather slots, and the active `ExecutionRuntime`.

Useful methods:

```java
context.set("key", value);
T value = context.get("key");
Optional<User> user = context.get("user", User.class);

context.setMetadata("schemaResult", result);
SchemaResult result = context.getMetadata("schemaResult");

context.memory("chat").append("user", input);
context.getLogger().info("message");

ExecutionRuntime runtime = context.getRuntime();
```

When to use:

Use `NodeContext` for execution-scoped state, metadata, logging, memory, and runtime access.

Be careful:

Do not use context as a hidden global data bus for everything. If a value is part of the workflow, prefer returning it from a node and passing it through edges. Use context for metadata, diagnostics, memory, and cross-cutting concerns.

---

### `Graph`

`Graph` stores nodes and edges. It is no longer a sequential list. It is an adjacency-map based DAG with explicit roots and terminals.

Important methods:

```java
Graph graph = Graph.builder("pipeline").build();

graph.getNodes();
graph.getAllEdges();
graph.getRootNodes();
graph.getTerminalNodes();
graph.findNodeByName("nodeName");
```

Definitions:

- Root node: a node with no non-cyclic incoming edge.
- Terminal node: a node with no outgoing edge.
- Multiple roots mean multiple entry points.
- Multiple terminal nodes mean `ParallelExecutor.run(...)` returns a map with one entry per terminal node.

When to use:

Most users should create graphs through `GraphBuilder`. Use direct `Graph` methods only for low-level graph construction or tooling.

---

### `Edge`

`Edge` connects a source node to a target node. It controls whether execution should move from source to target.

Core methods:

```java
public boolean shouldTraverse(Object output, NodeContext context) {
    return true;
}

public boolean shouldTraverseFailure(NodeFailure failure, NodeContext context) {
    return false;
}
```

When to use:

Use the built-in edge types first. Extend `Edge` only when you need a custom transition type that cannot be expressed with direct, conditional, failure, or cyclic edges.

## GraphBuilder DSL

`GraphBuilder` is the main DSL for creating graph workflows.

### 1. Named Graph

```java
Graph graph = GraphBuilder.named("support-ticket-pipeline")
    .addNode("classify", classifyNode)
    .addNode("route", routeNode)
    .connect("classify", "route")
    .build();
```

Alternative:

```java
Graph graph = Graph.builder("support-ticket-pipeline")
    .addNode("classify", classifyNode)
    .addNode("route", routeNode)
    .connect("classify", "route")
    .build();
```

When to use:

Use named graphs for production workflows. The graph name is stored in context metadata and helps logs and diagnostics.

---

### 2. Add Nodes

```java
Graph graph = GraphBuilder.named("pipeline")
    .addNode("normalize", new NormalizeText())
    .addNode("summarize", summarizeNode)
    .build();
```

`addNode(name, node)` wraps the node in a named wrapper. The graph name becomes the stable node identity even if the underlying class has a different `getName()`.

When to use:

Use graph-level names for stable workflow IDs. This is especially important when the same node class appears multiple times.

---

### 3. Add Parallel Root Nodes

```java
Graph graph = GraphBuilder.named("parallel-roots")
    .addParallelNodes(
        "extractTitle", titleNode,
        "extractTags", tagsNode,
        "extractPriority", priorityNode
    )
    .addNode("merge", merge)
    .connect("extractTitle", "merge")
    .connect("extractTags", "merge")
    .connect("extractPriority", "merge")
    .build();
```

When to use:

Use `addParallelNodes(...)` as a compact DSL when several independent nodes should start from the same graph input.

---

### 4. Direct Edges

```java
Graph graph = GraphBuilder.named("linear")
    .addNode("load", loadNode)
    .addNode("clean", cleanNode)
    .addNode("save", saveNode)
    .connect("load", "clean")
    .connect("clean", "save")
    .build();
```

When to use:

Use `connect(...)` for normal success paths.

---

### 5. Conditional Edges

```java
Graph graph = GraphBuilder.named("moderation")
    .addNode("classify", classifyNode)
    .addNode("allow", allowNode)
    .addNode("reject", rejectNode)
    .connectConditional("classify", "allow",
        (String label, NodeContext ctx) -> "SAFE".equals(label))
    .connectConditional("classify", "reject",
        (String label, NodeContext ctx) -> "UNSAFE".equals(label))
    .build();
```

When to use:

Use `connectConditional(...)` when branching is simple and belongs naturally to the edge.

Use `BranchNode` or `RouterNode` instead when:

- you want named route metrics,
- routing logic is substantial,
- the routing decision should be visible as a graph node,
- you need transform-per-route behavior.

---

### 6. Failure Edges

```java
Graph graph = GraphBuilder.named("fallback")
    .addNode("primary", primaryNode)
    .addNode("fallback", fallbackNode)
    .connectOnFailure("primary", "fallback")
    .build();
```

Failure edges are traversed only when the source node returns a `NodeFailure` under `ExecutionRuntime.FailureMode.COLLECT_ERRORS`.

```java
ExecutionRuntime runtime = ExecutionRuntime.builder()
    .failureMode(ExecutionRuntime.FailureMode.COLLECT_ERRORS)
    .build();

Map<String, Object> result =
    new ParallelExecutor(runtime).run(graph, input, new NodeContext());
```

When to use:

Use `connectOnFailure(...)` for recovery, fallback, error collection, and partial-failure joins.

---

### 7. Any-Failure Routing

Sometimes you know where failures should go, but you do not know which node will fail. Instead of writing one failure edge per node, use:

```java
Graph graph = GraphBuilder.named("global-failure-handler")
    .addNode("load", loadNode)
    .addNode("extract", extractNode)
    .addNode("validate", validateNode)
    .addNode("failureSink", failureSinkNode)
    .connect("load", "extract")
    .connect("extract", "validate")
    .connectAnyFailureTo("failureSink")
    .build();
```

Alias:

```java
.connectOnAnyFailure("failureSink")
```

This expands at build time into `FailureEdge`s from every registered node to the target, excluding the target itself. Explicit `connectOnFailure(from, to)` edges are preserved and not duplicated.

Failure-only outgoing edges do not hide a node's normal terminal output. A node that has no success-path outgoing edge is still treated as a terminal node even if it has an automatic failure edge.

With a merge node:

```java
MergeNode merge = new MergeNode.Builder()
    .expect("A", "B", "C")
    .build("merge");

Graph graph = GraphBuilder.named("auto-failure-merge")
    .addNode("A", nodeA)
    .addNode("B", nodeB)
    .addNode("C", nodeC)
    .addNode("merge", merge)
    .connect("A", "merge")
    .connect("B", "merge")
    .connect("C", "merge")
    .connectAnyFailureTo("merge")
    .build();
```

When to use:

Use `connectAnyFailureTo(...)` when a whole graph or subgraph has one common failure destination, such as:

- a fallback node,
- a dead-letter node,
- an audit/error collector,
- a `MergeNode` that should receive whichever branch failed.

Use explicit `connectOnFailure(from, to)` instead when different nodes need different failure recovery paths.

Runtime requirement:

Like explicit failure edges, any-failure routing requires `ExecutionRuntime.FailureMode.COLLECT_ERRORS` to continue after node failure and route the `NodeFailure`.

MergeNode note:

If the target is a `MergeNode`, its `.expect(...)` list still needs to include every node that may contribute success or failure. The any-failure DSL creates failure edges; it does not change merge expectations.

---

### 8. GraphBuilder Route DSL

`GraphBuilder.route(...)` creates conditional edges from one source node using a context-based router function.

```java
Graph graph = GraphBuilder.named("context-route")
    .addNode("start", startNode)
    .addNode("email", emailNode)
    .addNode("sms", smsNode)
    .route("start", ctx -> ctx.get("channel"))
        .to("email", "email")
        .to("sms", "sms")
        .end()
    .build();
```

When to use:

Use this DSL when the routing decision is stored in `NodeContext`, not directly in the node output.

Use `RouterNode` when the route should be computed from the input value and can fan out to multiple targets.

---

### 9. Repeat Same Node

```java
Graph graph = GraphBuilder.named("retry-same-node")
    .addNode("validate", validateNode)
    .repeat("validate")
        .whileCondition((output, ctx) -> !((Validation) output).isValid())
        .max(3)
        .build()
    .build();
```

When to use:

Use `repeat(nodeName)` when a node can re-run itself until a condition passes.

---

### 10. Loop To Another Node

```java
Graph graph = GraphBuilder.named("refine-loop")
    .addNode("draft", draftNode)
    .addNode("score", scoreNode)
    .connect("draft", "score")
    .loop("score")
        .to("draft")
        .whileCondition((output, ctx) -> ((Double) output) < 0.85)
        .max(5)
        .build()
    .build();
```

When to use:

Use `loop(from).to(target)` for refinement, repair, retry, or validation loops where control moves back to an earlier node.

Important:

Loops require `allowCycles()`. The builder calls it internally for `repeat(...)` and `loop(...)`. Cycles are only legal through `CyclicEdge`.

---

### 11. Explicit Graph API

```java
NodePlugin<String, String> a = new NormalizeText();
NodePlugin<String, String> b = new NormalizeText();

Graph graph = Graph.builder("explicit")
    .addNode("A", a)
    .addNode("B", b)
    .connect("A", "B")
    .build();
```

Most users should build graphs with the builder entry points:

```java
Graph.builder()
Graph.builder("name")
GraphBuilder.named("name")
```

`Graph` also has lower-level methods such as `addNode(...)`, `addEdge(source, target)`, and `addEdge(edge)`. Use those in framework code, graph tooling, tests, or when you need to work directly with explicit `Edge` objects.

When to use:

Use `GraphBuilder` for application workflows. Use the explicit `Graph` API only when building higher-level DSLs or graph tooling.

---

### 12. Explicitly Allow Cycles

`repeat(...)` and `loop(...)` call `allowCycles()` internally. If you add `CyclicEdge` manually, call it yourself:

```java
Graph graph = GraphBuilder.named("manual-cycle")
    .allowCycles()
    .addNode("A", nodeA)
    .addNode("B", nodeB)
    .connect("A", "B")
    .build();
```

When to use:

Only allow cycles when the graph has intentional bounded loops. Normal direct or conditional edges cannot form cycles.

## Edge Types

### `DirectEdge`

Always traverses after the source succeeds.

Builder DSL:

```java
.connect("A", "B")
```

Explicit:

```java
graph.addEdge(new DirectEdge(nodeA, nodeB));
```

When to use:

Use for normal linear pipelines.

---

### `ConditionalEdge<O>`

Traverses only when the predicate returns true.

```java
.connectConditional("classify", "urgent",
    (Ticket ticket, NodeContext ctx) -> ticket.priority().equals("HIGH"))
```

When to use:

Use when one node has output-dependent transitions and you do not need a dedicated routing node.

Be careful:

Multiple conditional edges from the same source can all fire. If you need exactly one path, use `BranchNode`.

---

### `FailureEdge`

Traverses only on failure.

```java
.connectOnFailure("callModel", "fallback")
```

For a graph-wide failure destination:

```java
.connectAnyFailureTo("fallback")
// or
.connectOnAnyFailure("fallback")
```

When to use:

Use for fallback, recovery, dead-letter, or error aggregation paths.

Use explicit `connectOnFailure(from, to)` for node-specific recovery. Use `connectAnyFailureTo(to)` when every node should share the same failure destination.

Runtime requirement:

Failure routing is meaningful with `ExecutionRuntime.FailureMode.COLLECT_ERRORS`. In `FAIL_FAST`, the graph stops on first failure.

---

### `CyclicEdge`

Creates an intentional loop with a maximum iteration count.

```java
.loop("validator")
    .to("generator")
    .whileCondition((output, ctx) -> !((ValidationResult) output).isValid())
    .max(3)
    .build()
```

When to use:

Use for retries, repair loops, self-correction, and iterative refinement.

Be careful:

Always keep `max(...)` small and explicit. Deterministic AI execution requires bounded loops.

---

## Execution

### `ParallelExecutor`

`ParallelExecutor` runs a `Graph` using `CompletableFuture`s and the active `ExecutionRuntime`.

```java
ParallelExecutor executor = new ParallelExecutor();
Map<String, Object> outputs = executor.run(graph, input, new NodeContext());
```

For graphs with exactly one terminal node:

```java
String output = executor.runSingle(graph, input, new NodeContext());
```

What `run(...)` returns:

- A `Map<String, Object>`
- Keys are terminal node names
- Values are terminal node outputs

When to use:

Use `ParallelExecutor` for graph execution, branching, fan-out, merge behavior, failure routing, and runtime-controlled concurrency.

---

### `ExecutionRuntime`

`ExecutionRuntime` controls shared execution behavior:

- executor service,
- max concurrency,
- global semaphore,
- default timeout,
- failure mode,
- metrics registry.

Example:

```java
ExecutionRuntime runtime = ExecutionRuntime.builder()
    .maxConcurrency(8)
    .failureMode(ExecutionRuntime.FailureMode.COLLECT_ERRORS)
    .defaultTimeout(30, TimeUnit.SECONDS)
    .build();

ParallelExecutor executor = new ParallelExecutor(runtime);
```

Failure modes:

| Mode | Behavior | Use when |
| --- | --- | --- |
| `FAIL_FAST` | Stop on first failure. | Partial results are not useful. |
| `COLLECT_ERRORS` | Continue and represent failures as `NodeFailure`. | Partial results are useful. |
| `SKIP_FAILED` | Continue but do not run downstream from failed node. | Failed branch should disappear. |

When to use:

Use a custom runtime whenever concurrency, failure behavior, or timeout semantics matter.

---

### `DAGValidator`

The executor validates graphs before running them.

```java
DAGValidator.validate(graph);

DAGValidator.ValidationResult result = DAGValidator.inspect(graph);
if (!result.isValid()) {
    System.out.println(result.summaryMessage());
}
```

Checks include:

- graph has nodes,
- graph has roots,
- graph has terminals,
- non-cyclic edges do not form cycles,
- unreachable/orphan nodes are warned.

When to use:

Use `inspect(...)` in tests, developer tooling, graph visualization, or CLI validation.

---

## Branching Nodes

### `BranchNode<I>`

`BranchNode` is an if / else-if / else node. It evaluates branches in order and returns the first match.

```java
BranchNode<String> branch = BranchNode.<String>builder()
    .when("short", text -> text.length() < 100)
        .then("summarizeSmall")
    .when("long", text -> text.length() >= 100)
        .then("summarizeLarge")
    .orElse("fallback")
    .build("branch");
```

Graph:

```java
Graph graph = GraphBuilder.named("branch-example")
    .addNode("branch", branch)
    .addNode("summarizeSmall", smallNode)
    .addNode("summarizeLarge", largeNode)
    .addNode("fallback", fallbackNode)
    .connect("branch", "summarizeSmall")
    .connect("branch", "summarizeLarge")
    .connect("branch", "fallback")
    .build();
```

The executor sees `BranchNode.RoutedResult` and schedules only the selected next node.

When to use:

Use `BranchNode` when exactly one path should run.

Do not use:

Do not use `BranchNode` for fan-out. If more than one path can run, use `RouterNode`.

---

### `RouterNode<I>`

`RouterNode` evaluates all routes and fires every matching route.

```java
RouterNode<String> router = RouterNode.<String>builder()
    .route("needsSummary", text -> text.length() > 500, "summary")
    .route("needsTags", text -> text.contains("#"), "tags")
    .route("alwaysAudit", text -> true, "audit")
    .requireAtLeastOne()
    .build("router");
```

With transforms:

```java
RouterNode<String> router = RouterNode.<String>builder()
    .route("toTitle", s -> true, s -> "title: " + s, "title")
    .route("toTags", s -> true, s -> "tags: " + s, "tags")
    .build("router");
```

When to use:

Use `RouterNode` for fan-out where multiple downstream nodes may run from one input.

Do not use:

Do not use `RouterNode` when only one route should run. Use `BranchNode` for mutually exclusive routing.

---

### `MergeNode`

`MergeNode` joins multiple upstream contributors. Each expected contributor must call into the merge through graph edges. It collects successes and failures.

```java
MergeNode merge = new MergeNode.Builder()
    .expect("A", "B")
    .timeout(2, TimeUnit.SECONDS)
    .strategy(MergeNode.MergeStrategy.COLLECT_ALL)
    .build("merge");
```

Graph:

```java
Graph graph = GraphBuilder.named("router-merge")
    .addNode("router", router)
    .addNode("A", nodeA)
    .addNode("B", nodeB)
    .addNode("merge", merge)
    .connect("router", "A")
    .connect("router", "B")
    .connect("A", "merge")
    .connect("B", "merge")
    .build();
```

Read the result:

```java
Map<String, Object> out = new ParallelExecutor().run(graph, input, new NodeContext());
MergeNode.MergeResult result = (MergeNode.MergeResult) out.get("merge");

Map<String, Object> successes = result.getSuccess();
Map<String, Throwable> errors = result.getErrors();
Object merged = result.getMerged();
```

Merge strategies:

| Strategy | Output from `getMerged()` | Use when |
| --- | --- | --- |
| `COLLECT_ALL` | `Map<String, Object>` | You want named branch outputs. |
| `FIRST_WINS` | First successful value by arrival order. | You race branches and take the fastest result. |
| `LIST` | `List<Object>` | You care about values, not contributor names. |

When to use:

Use `MergeNode` when graph-level branches must synchronize before downstream processing.

Be careful:

The names in `.expect(...)` must match upstream graph node names. Missing contributors cause `MergeTimeoutException`.

---

## Concurrency Nodes

### `ParallelNode<I, O>`

`ParallelNode` runs a fixed set of named tasks against the same input and returns a `ParallelResult`.

```java
ParallelNode<String, String> parallel = ParallelNode.<String, String>builder()
    .task("summary", text -> summarize(text))
    .task("keywords", text -> extractKeywords(text))
    .task("sentiment", text -> classifySentiment(text))
    .timeout(10, TimeUnit.SECONDS)
    .continueOnError()
    .build("parallelAnalysis");
```

Read the result:

```java
ParallelNode.ParallelResult<String> result =
    parallel.process("input text", context);

String summary = result.getOrDefault("summary", "");
boolean keywordsOk = result.succeeded("keywords");
Throwable sentimentError = result.getError("sentiment");
```

When to use:

Use `ParallelNode` when:

- you have a fixed known set of independent tasks,
- all tasks receive the same input,
- you want fan-out/fan-in hidden inside one node,
- the rest of the graph does not need to see individual task nodes.

Use `RouterNode` + `MergeNode` instead when:

- each branch is a real workflow step,
- branches have their own downstream paths,
- graph visualization should show the fan-out,
- failures need explicit graph routing.

---

### `MapNode<I, O>`

`MapNode` applies the same function to each element of an input `Iterable<I>` concurrently and returns a `MapResult<O>` in original input order.

```java
MapNode<String, String> summarizeBatch = MapNode.<String, String>builder()
    .mapWith(doc -> summarize(doc))
    .maxInFlight(8)
    .timeout(60, TimeUnit.SECONDS)
    .completionPollTimeout(5, TimeUnit.SECONDS)
    .continueOnError()
    .build("summarizeBatch");
```

Read the result:

```java
MapNode.MapResult<String> result =
    summarizeBatch.process(documents, context);

List<String> summaries = result.toSuccessfulList();
Set<Integer> failed = result.failedIndices();

for (int i : failed) {
    Throwable error = result.getError(i);
}
```

When to use:

Use `MapNode` for batch processing where the same operation is applied independently to many items.

Use `ParallelNode` instead when each task is different.

---

### `GatherNode`

`GatherNode` normalizes, filters, sorts, transforms, groups, and aggregates values. It accepts:

- `ParallelNode.ParallelResult`,
- `MapNode.MapResult`,
- `Iterable`,
- a single value.

Example:

```java
GatherNode gather = GatherNode.builder()
    .collectMode(CollectionMode.SUCCESS_ONLY)
    .filter((String s) -> !s.isBlank())
    .sortBy(Comparator.comparingInt(String::length))
    .limit(10)
    .transform((String s) -> s.toUpperCase())
    .aggregate(GatherNode.Aggregation.LIST)
    .build("gather");
```

Read the result:

```java
GatherNode.GatherResult result = gather.process(mapResult, context);

List<String> items = result.items();
List<String> value = result.value();
int count = result.count();
```

Collection modes:

| Mode | Meaning |
| --- | --- |
| `SUCCESS_ONLY` | Collect only successful values. |
| `FAILURES_ONLY` | Collect only failed task results. |
| `ALL_RESULTS` | Collect all `TaskResult` objects. |
| `COMPLETED_ONLY` | Collect successes and failures, not cancelled or not-executed. |

Aggregations:

| Aggregation | Result |
| --- | --- |
| `LIST` | List of final items. |
| `FIRST` | First item or null. |
| `LAST` | Last item or null. |
| `COUNT` | Item count. |
| `SUM` | Numeric sum. |
| `AVERAGE` | Numeric average. |
| `MAX` | Numeric max. |
| `MIN` | Numeric min. |

Group example:

```java
GatherNode grouped = GatherNode.builder()
    .groupBy((Ticket t) -> t.priority())
    .aggregate(GatherNode.Aggregation.LIST)
    .build("groupByPriority");
```

When to use:

Use `GatherNode` after `MapNode` or `ParallelNode` when downstream code should not care about the raw result shape.

Use `MergeNode` instead when you are joining separate graph branches by node name.

## Task Result Model

`ParallelNode` and `MapNode` use the sealed `TaskResult<T>` model:

| Class | Meaning |
| --- | --- |
| `Success<T>` | Task completed and produced a value. Null can be a valid success value. |
| `Failure<T>` | Task executed and threw an error. |
| `Cancelled<T>` | Task was submitted/running but cancelled. |
| `NotExecuted<T>` | Task was never submitted, usually because a deadline was reached. |

This is one of the important deterministic runtime pieces: failure is represented as data where possible.

When to use:

Use `TaskResult` when partial success matters. Avoid collapsing all batch work into a single exception if downstream nodes can still use successful items.

## Full Graph Example: Router + Merge + Failure Routing

```java
import java.util.Map;
import java.util.concurrent.TimeUnit;

import io.oxyjen.core.Graph;
import io.oxyjen.core.GraphBuilder;
import io.oxyjen.core.NodeContext;
import io.oxyjen.core.NodePlugin;
import io.oxyjen.execution.ExecutionRuntime;
import io.oxyjen.graph.ParallelExecutor;
import io.oxyjen.graph.branching.MergeNode;
import io.oxyjen.graph.branching.RouterNode;

class AppendNode implements NodePlugin<String, String> {
    private final String suffix;

    AppendNode(String suffix) {
        this.suffix = suffix;
    }

    @Override
    public String process(String input, NodeContext context) {
        return input + suffix;
    }
}

class FailingNode implements NodePlugin<String, String> {
    @Override
    public String process(String input, NodeContext context) {
        throw new RuntimeException("boom");
    }
}

RouterNode<String> router = RouterNode.<String>builder()
    .route("good", s -> true, "A")
    .route("bad", s -> true, "B")
    .build("router");

MergeNode merge = new MergeNode.Builder()
    .expect("A", "B")
    .timeout(2, TimeUnit.SECONDS)
    .build("merge");

Graph graph = GraphBuilder.named("collect-errors")
    .addNode("router", router)
    .addNode("A", new AppendNode("_A"))
    .addNode("B", new FailingNode())
    .addNode("merge", merge)
    .connect("router", "A")
    .connect("router", "B")
    .connect("A", "merge")
    .connectOnFailure("B", "merge")
    .build();

ExecutionRuntime runtime = ExecutionRuntime.builder()
    .failureMode(ExecutionRuntime.FailureMode.COLLECT_ERRORS)
    .build();

Map<String, Object> output =
    new ParallelExecutor(runtime).run(graph, "input", new NodeContext());

MergeNode.MergeResult result = (MergeNode.MergeResult) output.get("merge");

System.out.println(result.getSuccess()); // A -> input_A
System.out.println(result.getErrors());  // B -> RuntimeException("boom")
```

Why this is useful:

- `RouterNode` creates graph-level fan-out.
- `FailureEdge` sends failed branch `B` into the merge instead of losing it.
- `MergeNode` produces one explicit result containing both successes and errors.
- The runtime stays deterministic: failures are visible and structured.

## SchemaResult and SchemaNode

v0.5 also improves schema-driven LLM execution by using `SchemaResult` as the structured boundary between raw model output, validation, and graph decisions.

### `SchemaResult`

```java
public final class SchemaResult {
    private final String rawJson;
    private final boolean valid;
    private final List<FieldError> errors;

    public boolean isValid();
    public String getRawJson();
    public List<FieldError> getErrors();
}
```

Why it matters:

Older LLM workflows often return either a string or throw an exception. That makes graph-level decisions harder. `SchemaResult` keeps three pieces of information together:

- the raw JSON or raw model response,
- whether validation passed,
- the exact field-level errors if validation failed.

This lets Oxyjen support both hard-fail and soft-fail schema workflows.

---

### `SchemaNode<T>`

`SchemaNode<T>` combines:

- model call,
- JSON extraction,
- schema validation,
- retry prompt with validation errors,
- typed Java deserialization,
- optional memory storage,
- graph metadata for schema results and errors.

Example:

```java
record Ticket(String title, String priority) {}

JSONSchema schema = JSONSchema.object()
    .property("title", JSONSchema.PropertySchema.string("short title"))
    .property("priority",
        JSONSchema.PropertySchema.enumOf("priority", "LOW", "MEDIUM", "HIGH"))
    .required("title", "priority")
    .build();

SchemaNode<Ticket> node = SchemaNode.builder(Ticket.class)
    .model("gpt-4o-mini")
    .schema(schema)
    .maxRetries(3)
    .memory("chat")
    .failOnInvalid(true)
    .build();
```

Inside `process(...)`, `SchemaNode` stores:

```java
context.setMetadata("schemaResult", result);
context.setMetadata("schemaErrors", result.getErrors()); // only when invalid
context.setMetadata("deserializationError", e.getMessage()); // when mapping fails
```

When to use:

Use `SchemaNode<T>` when an LLM output becomes data for the graph. This is better than passing raw strings into downstream business logic.

---

### Hard-Fail Schema Mode

Default behavior is hard-fail:

```java
SchemaNode<Ticket> node = SchemaNode.builder(Ticket.class)
    .model(model)
    .schema(schema)
    .failOnInvalid(true)
    .build();
```

If validation fails after retries, the node throws. This is best when downstream nodes cannot safely continue without valid typed data.

When to use:

Use hard-fail for strict pipelines, persistence writes, tool arguments, payment actions, or any workflow where invalid data is dangerous.

---

### Soft-Fail Schema Mode

```java
SchemaNode<Ticket> node = SchemaNode.builder(Ticket.class)
    .model(model)
    .schema(schema)
    .failOnInvalid(false)
    .build();
```

In soft-fail mode:

- invalid schema output returns `null`,
- `schemaResult` is stored in metadata,
- `schemaErrors` is stored in metadata,
- downstream graph logic can inspect the error and route.

Example:

```java
Graph graph = GraphBuilder.named("schema-soft-fail")
    .addNode("extract", schemaNode)
    .addNode("repair", repairNode)
    .addNode("persist", persistNode)
    .connectConditional("extract", "persist",
        (Ticket ticket, NodeContext ctx) -> ticket != null)
    .connectConditional("extract", "repair",
        (Ticket ticket, NodeContext ctx) -> ticket == null)
    .build();
```

Inspect errors in a repair node:

```java
class RepairPromptNode implements NodePlugin<Object, String> {
    @Override
    public String process(Object input, NodeContext context) {
        SchemaResult result = context.getMetadata("schemaResult");
        return "Repair this JSON: " + result.getRawJson()
            + "\nErrors: " + result.getErrors();
    }
}
```

When to use:

Use soft-fail when the graph has a repair path, fallback model, human review step, or dead-letter queue.

### Pattern 1: LLM -> SchemaNode -> Branch

Use this when an LLM extracts structured data and the graph chooses one path.

```java
Graph graph = GraphBuilder.named("ticket-router")
    .addNode("extract", schemaNode)
    .addNode("branch", BranchNode.<Ticket>builder()
        .when("urgent", t -> "HIGH".equals(t.priority())).then("pageOnCall")
        .orElse("normalQueue")
        .build("branch"))
    .addNode("pageOnCall", pageNode)
    .addNode("normalQueue", queueNode)
    .connect("extract", "branch")
    .connect("branch", "pageOnCall")
    .connect("branch", "normalQueue")
    .build();
```

Why:

The LLM boundary is typed before routing. Branching logic is normal Java logic over a `Ticket`, not prompt text.

---

### Pattern 2: Router -> Parallel Branches -> Merge

Use this when multiple independent analyses should run and join.

```java
MergeNode merge = new MergeNode.Builder()
    .expect("summary", "sentiment", "risk")
    .build("merge");

Graph graph = GraphBuilder.named("analysis")
    .addNode("router", RouterNode.<String>builder()
        .route("summary", s -> true, "summary")
        .route("sentiment", s -> true, "sentiment")
        .route("risk", s -> true, "risk")
        .build("router"))
    .addNode("summary", summaryNode)
    .addNode("sentiment", sentimentNode)
    .addNode("risk", riskNode)
    .addNode("merge", merge)
    .connect("router", "summary")
    .connect("router", "sentiment")
    .connect("router", "risk")
    .connect("summary", "merge")
    .connect("sentiment", "merge")
    .connect("risk", "merge")
    .build();
```

Why:

The fan-out and fan-in are visible in the graph topology.

---

### Pattern 3: MapNode -> GatherNode

Use this for batch workflows.

```java
MapNode<String, String> map = MapNode.<String, String>builder()
    .mapWith(doc -> summarize(doc))
    .continueOnError()
    .build("summarize");

GatherNode gather = GatherNode.builder()
    .collectMode(CollectionMode.SUCCESS_ONLY)
    .aggregate(GatherNode.Aggregation.LIST)
    .build("successfulSummaries");

Graph graph = GraphBuilder.named("batch-summary")
    .addNode("summarize", map)
    .addNode("successfulSummaries", gather)
    .connect("summarize", "successfulSummaries")
    .build();
```

Why:

`MapNode` preserves per-item execution state. `GatherNode` turns that state into a clean downstream value.

---

### Pattern 4: FailureEdge -> MergeNode

Use this when partial results are valuable.

```java
ExecutionRuntime runtime = ExecutionRuntime.builder()
    .failureMode(ExecutionRuntime.FailureMode.COLLECT_ERRORS)
    .build();
```

```java
Graph graph = GraphBuilder.named("partial-failure")
    .addNode("A", nodeA)
    .addNode("B", nodeB)
    .addNode("merge", merge)
    .connect("A", "merge")
    .connectOnFailure("B", "merge")
    .build();
```

Why:

Failures become part of the merge result instead of disappearing or stopping the whole graph.

---

### Pattern 5: Bounded Repair Loop

Use this when a model output can be repaired.

```java
Graph graph = GraphBuilder.named("repair-loop")
    .addNode("generate", generator)
    .addNode("validate", validator)
    .connect("generate", "validate")
    .loop("validate")
        .to("generate")
        .whileCondition((output, ctx) -> !((SchemaResult) output).isValid())
        .max(3)
        .build()
    .build();
```

Why:

The loop is explicit and bounded. This is safer than an unbounded agent loop.

## Design Guidance

### Prefer Data Over Hidden Control Flow

Good:

```java
return new Classification("billing", 0.93);
```

Then route using Java predicates.

Avoid:

```java
context.set("nextNode", "billing");
return input;
```

Unless you are intentionally using the `GraphBuilder.route(...)` DSL.

---

### Keep Fan-In Explicit

If multiple branches must join, use `MergeNode`. Do not rely on side effects in `NodeContext` as an implicit join mechanism.

---

### Choose the Smallest Routing Primitive

- Simple output predicate: `ConditionalEdge`
- Exactly one branch: `BranchNode`
- Multiple branches: `RouterNode`
- Failure path: `FailureEdge`
- Loop: `CyclicEdge`

---

### Treat LLMs as Unreliable Boundaries

Use `SchemaNode<T>` or `SchemaEnforcer` when model output feeds the graph.

Use `SchemaResult` to preserve raw output plus validation errors.
